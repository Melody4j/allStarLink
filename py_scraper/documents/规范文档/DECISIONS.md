# 关键设计决策记录

本文档记录当前项目中已经达成一致的重要设计决策。

用途：

- 记录“为什么这么做”
- 避免未来重复讨论同一问题
- 让后续协作先读结论再改代码

建议后续新增决策时继续追加，保持编号递增。

## D-001 以 `src/link_scraper` 作为当前主线实现

状态：

- 已确认

决策：

- 当前项目应以 `src/link_scraper` 作为新版主线代码
- 历史代码和旧版脚本不作为当前实现依据

原因：

- 当前主要修复与演进都发生在 `src/link_scraper`
- 旧版实现与当前流程已经分叉

影响：

- 后续阅读、修复、重构优先基于 `src/link_scraper`

## D-002 旧 `documents` 中历史表结构说明不作为最终依据

状态：

- 已确认

决策：

- 历史文档如果与当前源码不一致，应以当前源码为准

原因：

- 旧文档中很多字段说明已经与当前代码、当前表结构偏离

影响：

- 反推真实结构时必须看当前代码
- 新文档应逐步替代旧文档

## D-003 子节点缺失统计值时不再写伪 `0`

状态：

- 已实施

决策：

- 子节点作为占位节点进入系统时：
  - `apprptuptime = null`
  - `total_keyups = null`
  - `total_tx_time = null`
  - `connections = null`

原因：

- `0` 会让查询结果产生误导
- 空值更准确表达“当前未知”

影响：

- 查询图数据时，需区分空值与真实零值
- 占位节点语义更清晰

## D-004 子节点更新 MySQL `dim_nodes` 时不覆盖 `current_link_count`

状态：

- 已实施

决策：

- 在主节点解析连接子节点时，更新子节点 `dim_nodes` 不应覆盖 `current_link_count`

原因：

- 子节点在该时刻只是被别人引用，并不代表已经拿到了它自身真实连接数

影响：

- `current_link_count` 只由节点列表扫描或主节点自身完整抓取决定

## D-005 ODS `links` 字段存储 `stats.data.nodes`

状态：

- 已实施

决策：

- `ods_nodes_details.links` 的业务来源固定为 `stats.data.nodes`

原因：

- 这是当前代码中真实的连接模式原始字符串

影响：

- 该字段不是数组语义，而是原始字符串语义

## D-006 ODS `links` 写库前转换为合法 JSON 字符串

状态：

- 已实施

决策：

- 由于数据库列类型为 JSON，`links` 在写库前必须做 `json.dumps()`

原因：

- 直接写入裸字符串会触发 MySQL JSON 校验错误

影响：

- 数据库存的是 JSON 字符串值
- 下游使用时应按该格式理解

## D-007 Neo4j 节点新增 `apprptuptime` 属性

状态：

- 已实施

决策：

- Neo4j `Node` 节点应写入 `apprptuptime`

原因：

- 在线时长属于节点重要统计属性，图查询时有使用价值

影响：

- 主节点完整抓取后会在图中保留在线时长

## D-008 更新子节点 Neo4j 数据时保留已有统计字段

状态：

- 已实施

决策：

- 子节点更新 Neo4j 时，不覆盖已存在的：
  - `total_keyups`
  - `total_tx_time`
  - `apprptuptime`

原因：

- 子节点占位数据不应覆盖主节点完整抓取得到的真实统计值

影响：

- 图节点统计字段更稳定
- 减少被占位数据回写污染的风险

## D-009 Neo4j 节点新增 `siteName` 属性

状态：

- 已实施

决策：

- Neo4j `Node` 节点增加 `siteName`

原因：

- `site_name` 是业务上常用的展示和分析字段

影响：

- 图查询中可以直接使用 `siteName`

## D-010 删除 `SnapshotScanner._cleanup_offline_nodes()` 废弃逻辑

状态：

- 已实施

决策：

- 移除 `SnapshotScanner` 中离线节点清理逻辑及其相关调用

原因：

- 该逻辑已被确认废弃
- 保留会引入无效代码和错误调用链

影响：

- 当前快照扫描器只负责：
  - 抓取节点列表
  - 更新连接数
  - 入 Redis 队列

## D-011 当前图模型采用按批次实例化节点

状态：

- 已确认

决策：

- Neo4j 节点唯一标识采用：

```text
{node_id}_{batch_no}
```

原因：

- 需要保留按批次划分的拓扑快照

影响：

- 同一业务节点在不同批次会出现多个图节点实例
- 查询必须明确批次范围

## D-012 当前架构不适合在不重构的情况下继续扩展多数据源

状态：

- 已确认

决策：

- 若后续要抓取其他网站，应先做结构化重构，而不是继续叠加现有代码

原因：

- 当前模型职责过重
- 当前主流程与 AllStarLink 字段结构耦合过深
- 当前 `APIWorker` 过于臃肿

影响：

- 多数据源接入前，应优先推进 source adapter 抽象和模型拆分

## D-013 重构优先级顺序

状态：

- 已确认

决策：

- 推荐的重构顺序为：

1. source adapter 抽象
2. 领域模型与持久化模型拆分
3. `APIWorker` 瘦身
4. repository 层收敛
5. 多数据源接入

原因：

- 这个顺序风险最低，且收益最大

影响：

- 后续重构工作应尽量遵循该顺序推进
