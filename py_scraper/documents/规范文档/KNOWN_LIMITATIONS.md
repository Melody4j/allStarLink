# 当前已知限制

本文档记录当前 `src/link_scraper` 在重构主体完成后的已知限制。

## 1. 当前范围限制

### 1.1 当前只保障 `AllStarLink`

当前真实实现、测试与联调都围绕：

- `AllStarLink`

展开。

当前虽然存在：

- `SOURCE_NAME`
- `source factory`
- `sources/other_source/`

但 `other_source` 仅是骨架，不代表第二数据源已经可运行。

### 1.2 当前不把第二数据源作为短期目标

在第二网站尚未明确之前，当前项目不继续推进：

- 第二数据源真实 client
- 第二数据源真实 parser
- 第二数据源真实 mapper

## 2. 模型层限制

### 2.1 旧兼容模型仍保留

当前 `src/link_scraper/models/` 仍存在，原因是：

- 需要兼容旧代码路径
- 需要兼容部分 manager 输入

影响：

- 项目内仍同时存在“旧模型”和“新模型”
- 阅读代码时要明确主流程优先看 `domain/` 和 `repositories/`

### 2.2 节点完整度仍未显式建模

当前子节点占位状态仍主要通过空值表达，例如：

- `apprptuptime is None`
- `total_keyups is None`
- `total_tx_time is None`
- `connections is None`

影响：

- 查询和调试时仍需要通过字段为空来推断“是否为占位节点”
- 这部分语义还没有收敛成正式字段

## 3. 架构层限制

### 3.1 兼容包装层仍存在

当前仍保留：

- `scrapers/node_parser.py`

它现在主要是兼容包装层，而不是主逻辑实现位置。

影响：

- 代码结构已经变新，但项目里仍会看到旧文件名

### 3.2 旧命名别名仍保留

当前主用命名已经切换为：

- `NodeIngestionWorker`
- `RelationalStorageManager`
- `GraphStorageManager`

但兼容别名仍保留：

- `APIWorker`
- `MySQLManager`
- `Neo4jManager`

影响：

- 外部引用暂时不会断
- 但命名体系还处于过渡期

## 4. 调度层限制

### 4.1 当前仍采用“队列清空后再扫描”策略

当前调度逻辑不是固定周期强制全量扫描，而是：

1. 队列清空
2. 触发新一轮快照扫描
3. 再继续消费队列

影响：

- 这是一种偏当前业务的调度方式
- 后续如果有不同数据源或不同抓取节奏需求，可能还要继续抽象调度层

## 5. 数据层限制

### 5.1 Neo4j 仍按批次实例化节点

当前唯一标识规则仍然是：

```text
unique_id = "{node_id}_{batch_no}"
```

影响：

- 同一业务节点在不同批次中会出现多个图节点实例
- 查询图数据时必须带上批次语义

### 5.2 `ods_nodes_details.links` 仍是“JSON 字符串值”

当前约定仍然是：

- 来源固定为 `stats.data.nodes`
- 写库前使用 JSON 序列化

影响：

- MySQL 中保存的是 JSON 字符串值
- 不是 JSON 数组

### 5.3 `dim_nodes` 仍然是选择性覆盖策略

当前 `dim_nodes` 写入仍然不是“全字段强覆盖”，而是按当前业务规则选择性更新。

影响：

- 阅读模型时不能假设所有字段都会无条件入库
- 实际行为仍要以 `DimNodeRecord` 和对应 SQL 为准

## 6. 验证层限制

### 6.1 已做联调，但不是全场景压测

当前已经完成一轮真实环境联调，验证了：

- 初始化
- 快照扫描
- 队列入队
- 单轮详情消费

但当前还没有覆盖：

- 长时间稳定性
- 高并发压测
- 大批量批次连续运行验证

## 7. 当前默认有效的设计结论

以下结论当前默认成立：

1. `src/link_scraper` 是当前主线代码
2. 当前主线只关注 `AllStarLink`
3. 子节点缺失统计数据时写 `null`，不写伪 `0`
4. `ods_nodes_details.links` 记录 `stats.data.nodes`，写库前转成 JSON 字符串
5. Neo4j 继续采用批次快照建模
6. 第二数据源只保留骨架，不做真实实现

## 8. 后续优先建议

如果后续继续推进，优先建议处理：

1. 显式建模节点完整度状态
2. 逐步移除旧兼容模型与旧兼容别名
3. 视变更范围继续补充集成验证
4. 当第二数据源确定后，再推进 `other_source` 的真实实现
